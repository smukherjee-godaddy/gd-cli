DEVX-70: Surface `app-registry-api` errors with accurate classification and actionable fix hints across every app-lifecycle command (create, update, archive, deploy upload). Server-classified failures — `VALIDATION_ERROR` (now forwarding per-field details), `AUTH_ERROR` / `UNAUTHORIZED`, `FORBIDDEN` / `INSUFFICIENT_PERMISSIONS`, `NOT_FOUND` (and known `*_NOT_FOUND` variants), `DUPLICATE_CONFLICT` / `STATE_CONFLICT` / `VERSION_CONFLICT`, and `RATE_LIMIT_EXCEEDED` — now map to a single `ServerError` tagged error with a `kind` discriminant (`"VALIDATION" | "FORBIDDEN" | "NOT_FOUND" | "CONFLICT" | "RATE_LIMITED"`), so the JSON envelope reports an accurate top-level `code` and a specific `fix` instead of collapsing everything into `NETWORK_ERROR`. `ValidationError` is now reserved for client-side (arktype) input failures only — "we rejected your input before sending" is semantically different from "the server rejected your request." Unknown / future server codes and transport failures still map to `NetworkError`, but now preserve the full HTTP context (status + response body) so agents can still discriminate via `error.details.response.errors[].extensions.code`. Classification is driven by an explicit code map in the shared `src/services/graphql-error.ts` module (used by both the application service and the extension presigned-URL service); unknown codes fall through to `NetworkError` rather than being captured by lexical heuristics.
